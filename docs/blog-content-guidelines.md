# Ainnc 博客内容长期规范

这份规范以 2026 年 7 月 10 日确认的《麦芽营销 Ainnc 博客文章编辑要求（完整版）》为唯一权威版本。新文章、历史文章修订和构建检查都必须遵守，不再沿用“整篇控制在 500–800 字”或“单段不超过 320 字”的旧标准。

## 1. 文章要先有判断，不写说明书

- 每篇文章必须提出一个明确观点或反直觉洞察，不能只复述产品功能和行业常识。
- 文章可以从具体场景、反例、数据异常、对话、结论或一个未解决的问题切入；选择最适合该主题的进入方式，不把“场景—原因—方法”固化成唯一顺序。
- 不用“第一层、第二层、第三层”或“A 是、B 是、C 是”撑满段落；分类可以保留，但必须服务于推理。
- 读出声像产品规格、FAQ 条目或定义合集的段落，需要重写成叙述和因果关系。

## 2. 硬性长度与内容密度

- 除列表、表格、引用和 FAQ 外，每个正文自然段不少于 400 字且不少于 4 句话。
- 每个正文段落必须包含数字、具体场景、例子或明确判断依据，不能换一种说法重复同一个结论。
- 每个主要二级标题下至少有一段完整论述；行动清单与 FAQ 模块可以使用短内容。
- 引用真实客户数据必须脱敏；假设场景必须明确使用“比如、例如、假设、具体”等提示。

## 3. 文章结构按内容决定

- TL;DR、引用块、二级标题、表格、流程图、代码块、行动清单和 FAQ 全部是可选元素，不设默认组合。
- 纯叙述推理可以不使用任何小标题；分场景、时间线、正反对照、案例复盘和问题递进也都可以成为正文骨架。
- 表格只用于三项以上确实需要逐字段比较的信息；流程图用于存在前后依赖的步骤；代码块用于必须精确保留的参数、记录格式或配置。
- 使用某个结构元素前先问：删除它以后，内容是否仍然能清楚表达？如果可以，就不要为了版式完整硬加。
- 每篇文章必须以完整自然段结束。清单、表格、代码、FAQ、引用或产品介绍都不能充当最后一个内容块。
- 收尾要回应开头的核心问题、给出最终判断，或说明方法背后的真正价值；不能批量套用“总结”“结语”“写在最后”等通用标题。
- 连续发布文章要比较结构指纹；相邻 5 篇若骨架相同，必须在发布前重新设计。
- FAQ 只用于真实搜索问题、选型疑问或正文难以自然展开的关键边界；不能把标题改成问句，也不能重复正文结论。
- 保留 FAQ 的文章必须在 frontmatter 写明 `faqIntent`；全库 FAQ 使用比例不得超过 15%。

## 4. 语言与品牌语气

- 使用人话，句子可以有节奏变化，但不能连续堆砌短定义句。
- 禁用或改写“依赖关系、执行前提、颗粒度、抓手、赋能、闭环、链路、生态、体系化、精细化运营”等分析腔词汇。
- 产品能力只能解释它如何帮助当前问题，不能让文章变成功能清单。
- 不写无法证明的“最好、绝对安全、行业领先”等结论。
- 比喻和金句必须帮助读者理解和复述，而不是只追求夸张。

## 5. 敏感话题与案例

- 限流、封号和风控只讨论合规、暂停、排查和复盘，不提供规避平台检测的技巧。
- 客户案例必须脱敏，案例只承担论据作用，不能占据整篇文章。
- 涉及平台规则时避免永久性保证，明确账号阶段、时间范围和观察条件。

## 6. 发布前强制步骤

1. 运行 `npm run verify:blog`，逐篇检查段落字数、句数、重复、具体信息、说明书腔与文章结构。
2. 对脚本标记的说明书腔段落重新阅读并人工判断，不能只为了通过正则替换几个词。
3. 校验失败时修改文章，不得降低脚本门槛迁就内容。
4. 运行 `npm run build`，确认 Astro 能解析所有内容并生成完整页面。
5. 实际打开文章页，检查长段落、可选结构元素、真实收尾和移动端可读性；不要把“模块齐全”误认为内容完整。

校验脚本位于 `scripts/verify_blog_length.py`，并已接入 `prebuild`。以后新增文章如果不符合本规范，正式构建会直接失败。
